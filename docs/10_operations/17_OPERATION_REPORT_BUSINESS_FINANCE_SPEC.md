# Операция «Отчёт по проекту» — бизнес-постановка: суммы, статья расходов, кошельки и жизненный цикл

**Назначение документа:** дать другой команде **полное бизнес-ТЗ** для воспроизведения процесса «отчёт о выполненных работах / расходах по проекту» с теми же **финансовыми потоками** и **правилами согласования**, но на **своей архитектуре** и со **своими именами сущностей в коде и БД**.  

**Имена таблиц, колонок и перечислений из исходной системы намеренно не используются** — ниже только бизнес-смысл, роли и правила.

**Референс-реализация (GURU, для сверки логики, не для копирования имён):** `docs/10_operations/16_OPERATION_REPORT.md` и связанные модульные документы по статьям расходов и прайс-листам.

---

## 1. Цель операции

**Отчёт** фиксирует факт: по проекту выполнены работы (или иные позиции), за которые **заказчик проекта** принимает обязательство оплатить **итоговую сумму к оплате заказчиком**, а **исполнитель (основной получатель по отчёту)** получает **сумму к выплате исполнителю**. Разница между тем, что заказчик платит **за базу работ** (без учёта наценки статьи) и тем, что уходит исполнителю, трактуется как **прибыль проекта**, распределяемая по правилам **статьи расходов**. **Наценка статьи** (если включена) добавляется к базе при определении **итога к оплате заказчиком** и распределяется отдельным набором долей.

Операция проходит **согласование** и в определённые моменты **проводится по кошелькам участников проекта** (см. разделы 6–8).

---

## 2. Участники и объекты (бизнес-смысл)

| Понятие | Смысл |
|--------|--------|
| **Проект** | Контекст; у проекта есть заказчик, руководитель, другие участники, кошельки. |
| **Инициатор отчёта** | Участник проекта, который создал черновик и отправляет его в согласование. |
| **Руководитель проекта** | Участник с полномочиями подтверждать отчёт на стороне исполнителя (до заказчика), завершать период ожидания после заказчика, откатывать завершённый отчёт (если продуктом предусмотрено). |
| **Технадзор** | Опциональная роль на проекте; если активна по правилам продукта, отчёт не идёт сразу к руководителю, а проходит этап технадзора. |
| **Заказчик проекта** | Участник с ролью заказчика; **не может** совпадать с основным получателем суммы по отчёту. Подтверждает или отклоняет отчёт с точки зрения оплаты **итога к оплате заказчиком**. |
| **Основной получатель (исполнитель по отчёту)** | Контрагент/участник, на которого начисляется **сумма к выплате исполнителю**; выбирается при заполнении отчёта. |
| **Статья расходов проекта** | Справочник внутри проекта: задаёт, включена ли **наценка** и какой у неё процент, а также **доли распределения прибыли** и (если наценка включена) **доли распределения наценки** между контрагентами/участниками (в процентах, сумма долей по каждому типу = 100%). |
| **Строки отчёта** | Перечень позиций: наименование, единица, количество, цена за единицу «исполнителю», цена за единицу «заказчику», итоги по строке. Источник строки: либо из прикреплённого к проекту прайс-листа, либо произвольная (ручная) позиция. После сохранения отчёта строки — **снимок**: изменение прайса позже **не пересчитывает** уже сохранённый отчёт. |
| **Кошелёк участника проекта** | У каждого участника проекта есть учёт денег в разрезе проекта. Для целей отчёта используются **два разных показателя** (см. раздел 6). |

## 3. Расчёт сумм на отчёте (до проведения по кошелькам)

Все денежные величины ведут с **фиксированной точностью** (как минимум до копеек); распределение процентов по долям — с правилом «целые минимальные единицы учёта» (например, центы) и разнесением остатка по очереди, чтобы сумма долей совпала с распределяемой суммой.

1. **Сумма к выплате исполнителю** = сумма по всем строкам отчёта полей «итог исполнителю по строке».
2. **База к оплате заказчиком** = сумма по всем строкам отчёта полей «итог заказчику по строке» (до наценки статьи).
3. **Сумма наценки** = если в статье расходов наценка **выключена** — ноль. Если **включена** — база к оплате заказчиком × процент наценки из статьи (по правилам округления продукта).
4. **Итог к оплате заказчиком** = база к оплате заказчиком + сумма наценки.
5. **Прибыль по отчёту** = база к оплате заказчиком − сумма к выплате исполнителю.  

   **Ограничение MVP:** отрицательная прибыль **запрещена** — отчёт нельзя сохранить/отправить, если исполнителю суммарно больше, чем заказчик платит за базу работ.

**Важно для кошельков:** при проведении отчёта с **кошелька заказчика** списывается именно **итог к оплате заказчиком**, а не только база. Наценка **не увеличивает** сумму, идущую исполнителю напрямую: она распределяется по **долям наценки** статьи на **показатель личного начисленного дохода** у выбранных участников (см. ниже).

---

## 4. Статья расходов: связь с отчётом

- Каждый отчёт привязан к **одной активной статье расходов** проекта.
- У статьи заданы:
  - **Доли прибыли:** кому из контрагентов (участников) какой процент от **прибыли по отчёту** начислить на **личный заработок** после проведения.
  - **Наценка:** признак «включена/выключена», процент; при включении — **доли наценки** (кому какой процент от **суммы наценки** начислить на **личный заработок**).
- Суммы долей по каждому типу (прибыль / наценка) в сумме дают 100% с допустимой точностью продукта (например, 33,33 + 33,33 + 33,34).

Участник для начисления по доле **находится** по контрагенту из настройки доли в рамках проекта; если нужного участника ещё нет (например, контрагент указан в доле, но не в списке участников), продукт может **создавать участника второго порядка** по правилам домена (как в референс-системе).

## 5. Кошелёк участника: какие показатели трогает отчёт

В учёте участника проекта различают как минимум:

| Показатель (бизнес-смысл) | Роль в отчёте |
|---------------------------|----------------|
| **Подотчётный баланс** | Отражает обязательства заказчика перед проектом «с точки зрения лимита / счёта заказчика»; при проведении отчёта **уменьшается** на величину **итога к оплате заказчиком** **только у заказчика проекта**. |
| **Личный заработок (начисленный доход участника)** | **Увеличивается** у основного получателя на **сумму к выплате исполнителю**; **увеличивается** у каждого бенефициара **долей прибыли** на его долю от прибыли; **увеличивается** у каждого бенефициара **долей наценки** на его долю от суммы наценки (если наценка > 0). |

Другие показатели кошелька отчёт **не меняет** (в референс-реализации разрешены только эти два типа движения).

---

## 6. Журнал проводок отчёта (для отката)

При каждом изменении кошелька из-за отчёта система должна **запоминать**:

- какой отчёт;
- какой участник проекта;
- какой показатель кошелька (подотчётный баланс или личный заработок);
- знак и величину изменения в минимальных единицах учёта;
- время проведения.

**Откат** при отказе заказчика или продуктовом «откате завершённого» выполняется **строго по сохранённым записям проводок** (обращение знака по каждой неоткаченной строке), а не пересчётом «как бы сейчас» из текущих строк отчёта или прайса.

## 7. Когда деньги по отчёту реально двигаются (проведение)

Ключевой принцип: **проведение по кошелькам не обязано совпадать с моментом «отправил черновик»** — оно зависит от того, **кто инициатор** и от наличия **технадзора**.

### 7.1. Отправка черновика в согласование

- Если **инициатор — руководитель проекта:** при первой отправке из черновика система **сразу проводит** отчёт по кошелькам (все движения из раздела 8), затем переводит процесс в состояние **«ожидает решения заказчика»**.
- Если **инициатор не руководитель** и на проекте **есть активный технадзор:** отправка ведёт в **«ожидает технадзора»** — **без** проведения по кошелькам.
- Если **инициатор не руководитель** и технадзора **нет:** отправка ведёт в **«ожидает руководителя проекта»** — **без** проведения по кошелькам.

### 7.2. Решение технадзора

- Одобрение: переход к **«ожидает руководителя проекта»** — деньги **ещё не** двигались, если не было проведения на шаге 7.1.
- Отклонение: возврат в **черновик** — проведения нет.

### 7.3. Подтверждение руководителем проекта (когда отчёт в состоянии «ожидает руководителя»)

- Если проведение **ещё не выполнялось** — система **выполняет проведение** (раздел 8), затем переводит в **«ожидает решения заказчика»**.
- Если проведение **уже было** (случай инициатора-руководителя) — повторного проведения нет, статус просто меняется дальше по сценарию (в референсе при подтверждении руководителем после технадзора проведение делается один раз здесь).

### 7.4. Отклонение руководителем проекта из «ожидает руководителя»

- Возврат в **черновик**. В штатном сценарии на этом этапе проведение **ещё не выполнялось** (оно выполняется при **подтверждении** руководителем, если инициатор не был руководителем), либо деньги уже **откатаны** при возврате от заказчика (см. §7.6) — дополнительный откат при отказе руководителя **не требуется**. Целевая система должна гарантировать согласованность: не оставлять «висящие» проводки без явной политики.

### 7.5. Подтверждение заказчиком

- Проведение **не повторяется**: деньги уже списаны у заказчика и начислены получателю и держателям долей **на шаге 7.1 или 7.3**.
- Переход в состояние **«ожидание завершения периода»** (например, 24 часа для отмены заказчиком — если заложено в продукте); фиксируется время начала этого периода.

### 7.6. Отклонение заказчиком (из «ожидает решения заказчика»)

- Если проведение **было** — выполняется **полный откат** всех движений по журналу проводок этого отчёта; затем процесс возвращается к **«ожидает руководителя проекта»** (черновик с точки зрения заказчика снят, но отчёт снова требует стороны исполнителя).
- Обязательный **комментарий** при отклонении.

### 7.7. Завершение после ожидания

- **Ручное завершение** руководителем из периода ожидания или **автоматическое** по истечении таймера: статус **«завершён»**, **без** новых движений по кошелькам (они уже были на этапе до заказчика).

### 7.8. Откат завершённого отчёта (если продуктом разрешён)

- Выполняется **тот же** откат по журналу проводок, что и при отказе заказчика после проведения; статус возвращается к **«ожидает руководителя проекта»**; обязателен комментарий.

## 8. Состав одной проводки отчёта (последовательность движений)

Все шаги в **одной транзакции**, с блокировками сущностей «отчёт / кошелёк», чтобы не применить проведение дважды.

1. **Заказчик:** уменьшить **подотчётный баланс** на **итог к оплате заказчиком**.
2. **Основной получатель:** увеличить **личный заработок** на **сумму к выплате исполнителю**.
3. **Доли прибыли:** для каждой строки настройки статьи с ненулевой суммой — найти участника проекта по контрагенту доли, увеличить его **личный заработок** на выделенную часть **прибыли по отчёту** (сумма частей = прибыль в минимальных единицах).
4. **Доли наценки** (если сумма наценки > 0): аналогично — распределить **сумму наценки** по долям на **личный заработок** участников.
5. Зафиксировать факт «проведение выполнено» и записать все строки журнала с временем проведения.

---

## 9. Жизненный цикл (цепочка состояний, бизнес-язык)

Ниже — логическая цепочка; точные имена статусов в целевой системе могут быть любыми, важна **семантика**.

| Состояние | Кто двигает | Проведение по кошелькам |
|-----------|-------------|-------------------------|
| Черновик | Инициатор редактирует | Нет |
| Ожидает технадзора | Технадзор | Нет |
| Ожидает руководителя проекта | Руководитель | См. §7.3 — **да**, если ещё не проводили |
| Ожидает заказчика | Заказчик | Уже проведено, если прошли шаги §7.1 или §7.3 |
| Ожидание периода (например, 24 ч) | Таймер или руководитель | Нет новых движений |
| Завершён | Система / руководитель | Нет новых движений |

Ветки отклонений: технадзор → черновик; руководитель из своего ожидания → черновик; заказчик после проведения → откат + снова «ожидает руководителя»; завершённый → откат по политике продукта.

---

## 10. Инварианты и правила согласованности

1. **Заказчик ≠ основной получатель** по одному отчёту.
2. **Итог к оплате заказчиком** — единственная величина списания с **подотчётного баланса заказчика** при проведении.
3. **Сумма к выплате исполнителю** — единственная прямая выплата на **личный заработок** основного получателя при проведении.
4. Распределение **прибыли** и **наценки** — только на **личный заработок**, не на подотчётный баланс.
5. **Повторное проведение** одного и того же отчёта запрещено: перед проверкой система должна убедиться, что проведение ещё не выполнялось.
6. **Откат** всегда опирается на **журнал проводок**, а не на пересчёт из актуального прайса или изменённых строк.

---

## 11. Что должна уметь целевая система для полного воспроизведения

- Модель **отчёта** с привязкой к проекту, статье расходов, заказчику, исполнителю, инициатору, дате операции, комментариям и **набору строк-снимков**.
- Расчёт **пяти ключевых сумм** (раздел 3) с запретом отрицательной прибыли в MVP.
- Настройка статьи расходов: **доли прибыли**, **наценка и доли наценки**.
- Учёт **двух показателей** на кошельке участника и строгое правило, какой показатель меняет отчёт.
- **Журнал проводок** и идемпотентность проведения / полный откат.
- **Жизненный цикл** с ветвлением «инициатор = руководитель» vs «не руководитель + технадзор» vs «не руководитель без технадзора», включая момент первого проведения и откат при отказе заказчика после проведения.
- (Опционально продуктом) **Связь отчёта с отдельной операцией «перевод денег»** и запрет дублирования одного перевода в двух отчётах — если в целевой системе тот же сценарий.

---

## 12. Сводка финансовых потоков (одним абзацем)

При первом допустимом по жизненному циклу **проведении** система **уменьшает подотчётный баланс заказчика** на **итог к оплате заказчиком** (база работ у заказчика + наценка статьи, если включена), **увеличивает личный заработок исполнителя** на **сумму к выплате исполнителю**, затем **разносит прибыль** (база заказчика минус выплата исполнителю) по **долям прибыли** статьи и **разносит наценку** по **долям наценки** на **личный заработок** указанных участников. Любое последующее **отменение проведённого** отчёта по правилам продукта **обращает** все эти движения по сохранённому журналу.
