## Уточнения к ТЗ-10A (решения)

### 1. Точность процентов

```text
2 знака после запятой
минимальный шаг = 0.01%
```

Валидация строгая:

```text
sum(profit_shares.percent) = 100.00
sum(markup_shares.percent) = 100.00, если markup_enabled = true
```

Комбинация допустима:

```text
33.33 + 33.33 + 33.34 = 100.00
```

Погрешность сверх нормализации до 2 знаков не закладывать.

---

### 2. PATCH вложенных массивов

`profit_shares` и `markup_shares` при PATCH считаются **полной заменой набора строк**.

При сохранении:

```text
старые строки долей удаляются / заменяются
новые строки записываются заново
всё внутри одной транзакции
```

Частичное слияние по id строки доли в MVP не делается.

---

### 3. Маршрут `/recipients`

Маршрут:

```http
GET /projects/{projectId}/expense-items/recipients
```

объявляется **раньше** параметрического:

```http
GET /projects/{projectId}/expense-items/{expenseItemId}
```

Для `{expenseItemId}` использовать **числовое** ограничение; строка `recipients` не может быть id статьи.

---

### 4. `is_active` и SoftDeletes

Используются оба механизма:

```text
is_active = false
deleted_at = now()
```

- `is_active` — бизнес-флаг активности.
- `deleted_at` — технический soft-delete Laravel.

Для обычного списка активных статей:

```text
is_active = true
deleted_at = null
```

---

### 5. Удалённые статьи и редактирование

В MVP восстановление и редактирование удалённой статьи **не делаются**.

Для пользователя удаление окончательное:

```text
статья исчезает из интерфейса
в новых отчётах недоступна
```

Технически запись сохраняется для исторических отчётов.

---

### 6. Заглушки «прибыль / наценка / сумма» на карточке

В MVP расчётные блоки как placeholder **не показываются**.

На карточке только:

```text
название статьи
наценка включена / выключена
процент наценки, если включена
количество получателей прибыли
количество получателей наценки
```

Расчётные показатели появятся позже вместе с REPORT.

---

### 7. Роли и навигация

Для ролей **без** права просмотра пункт **«Статьи расходов»** в проекте **не показывается**.

Не видят раздел:

```text
SUPERVISOR
EMPLOYEE
CUSTOMER
SUPPLIER
CONTRACTOR
second-order participants
```

Видят:

```text
OWNER — управление
PROJECT_HEAD — управление
PARTNER first-order — только просмотр
```

---

### 8. Множественный выбор получателей

Множественный выбор означает:

```text
пользователь выбирает нескольких контрагентов
нажимает «Подтвердить»
они добавляются в блок долей
проценты по каждому вводятся вручную после добавления
```

Автоматически проценты не распределять.

---

### 9. Выбор получателей в UI (MVP)

В клиенте **нет вкладок** «Участники проекта / Контрагенты компании»: один экран (bottom sheet) — **список активных контрагентов компании**, **поиск** по ФИО/email, **множественный выбор**, после подтверждения строки попадают в блок долей; **проценты** задаются вручную на форме статьи.

API **`GET …/expense-items/recipients`** в MVP всегда возвращает тот же набор (**контрагенты компании**); поле **`source`** в ответе зафиксировано как **`company_counterparties`** (расширение фильтров — позже).

При создании статьи расходов **`ProjectParticipant` не создаётся**; хранится только **`counterparty_id`**. Проверка «есть ли контрагент в проекте как first-order / second-order участник» выполняется **на этапе REPORT**, не в ТЗ-10A.
