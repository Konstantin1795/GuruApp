Документ нужен, чтобы Cursor понимал направление, но не принимал будущие правила за текущий код.

---

## 2. Назначение

`REPORT` должен фиксировать расход подотчётных средств, начислять заработанное и влиять на аналитику проекта.

---

## 3. Предварительные участники

Возможные участники процесса:

```text
Инициатор: EMPLOYEE / PARTNER / PROJECT_HEAD
Технадзор: SUPERVISOR, если назначен
Руководитель проекта: PROJECT_HEAD
Заказчик: CUSTOMER
```

---

## 4. Предварительный lifecycle

Черновая цепочка:

```text
CREATED
→ SUPERVISOR_APPROVAL, если нужен
→ PROJECT_HEAD_APPROVAL
→ CUSTOMER_APPROVAL
→ WAITING_24_HOURS
→ COMPLETED
```

Детали переходов, применения финансов и ветки **SUPERVISOR** — см. **раздел 9** и **ТЗ-10C**.

---

## 5. Финансовая логика (устаревшие абзацы заменены уточнениями)

Ранее в этом черновике фигурировало уменьшение подотчёта **исполнителя** — **не использовать**: в актуальной модели REPORT **не меняет подотчётные поля основного получателя отчёта** (`accountable_balance` / `accountable_spent` / `accountable_received` у основного получателя).

Итоговая математика дельт, заказчик, доли прибыли/наценки, откат по `report_wallet_deltas` и запрет отрицательной прибыли в MVP — см. **раздел 9** и полный текст **ТЗ-10C** (вне репозитория у постановщика).

---

## 6. Связь с переводами

Отчёт может создавать или подготавливать связанные операции `TRANSFER`.

Важно:

```text
переводы живут своим lifecycle
откат отчёта не должен автоматически ломать lifecycle переводов без отдельного правила
```

## 9. Актуальные уточнения перед реализацией ТЗ-10C

Согласовано перед стартом кода (приоритет при конфликте: последние уточнения в чате → **ТЗ-10C** (`TZ_10C_GURU_Report_Foundation.md`) → актуальные `docs/10_operations/*.md` → старые внешние ТЗ).

1. REPORT реализуется через отдельную таблицу **`report_operations`**, без общей таблицы **`operations`** для бизнес-логики REPORT.
2. REPORT добавляется в агрегированную историю как третий **`operation_kind = REPORT`** рядом с TRANSFER и INCOME.
3. Для REPORT нужен **`ReportAvailableActionsService`**.
4. **CUSTOMER** проекта — first-order участник проекта и **не равен** основному получателю отчёта.
5. Основной получатель отчёта **не может быть** Заказчиком проекта.
6. Если получателя отчёта нет в проекте — создать second-order **`ProjectParticipant`** + кошелёк.
7. **SUPERVISOR** определяется как активный **`ProjectParticipant`**: `project_role_code = SUPERVISOR`, `level = first`, `is_active = true`.
8. Отрицательная прибыль `profit_amount < 0` в MVP **запрещена**, возвращать **422** до применения финансов.
9. **WAITING_24_HOURS** считается по `waiting_period_started_at` + 24 часа в UTC / серверной зоне, **как у прочих операций**.
10. Для REPORT нужна отдельная **scheduled command/job** завершения 24 часов (в стиле проекта).
11. Редактирование строк/сумм разрешено **только до применения финансов** или **после отката** финансов.
12. После применения финансов в **CUSTOMER_APPROVAL** / **WAITING_24_HOURS** / **COMPLETED** редактирование строк/сумм **запрещено** (исключение: возврат в **PROJECT_HEAD_APPROVAL** после customer reject или COMPLETED rollback **и** дельты откатаны — РП может править и снова отправить заказчику).
13. Откат **COMPLETED → PROJECT_HEAD_APPROVAL** откатывает финансы и делает отчёт **pending** для РП.
14. **OWNER** может создать TRANSFER к REPORT **только если** он является участником проекта.
15. Если OWNER — участник проекта, но не **PROJECT_HEAD**, TRANSFER идёт по **partner-like** lifecycle (при необходимости с веткой SUPERVISOR по тем же правилам, что для переводов к отчёту).
16. Для REPORT и TRANSFER нужен **`operation_number`**: **`REP-{id}`** / **`TRF-{id}`**, с padding в UI при отображении.
17. Для REPORT нужна таблица **`report_wallet_deltas`**; откат финансов выполняется **строго по сохранённым дельтам**, не пересчётом из текущих строк/справочников.
18. Вкладка **«Переводы к отчёту»** в MVP **рабочая**, а не заглушка.
19. Заказчик **не видит** вкладку **«Переводы к отчёту»**.
20. В отчёте есть кнопка **«Прикрепить перевод»**, в переводе — **«Прикрепить к отчёту»**.
21. Поиск привязки выполняется по **`operation_number`**, дате и сумме.
22. В одном отчёте можно использовать позиции из **разных** прикреплённых прайс-листов и **custom**-позиции.
23. Строки отчёта хранят **полный snapshot** и не пересчитываются после изменения прайсов.
24. REPORT **не меняет** подотчётные поля **основного получателя** отчёта.
25. При конфликте приоритет: последние уточнения пользователя в чате → ТЗ-10C → актуальные `docs/10_operations/*.md` → старые ТЗ.
