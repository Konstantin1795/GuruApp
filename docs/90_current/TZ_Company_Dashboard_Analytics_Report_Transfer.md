# ТЗ — Главный аналитический виджет GURU: доход, задолженность, переплата, месяцы и списки операций

## 1. Назначение задачи

Нужно доработать математику и поведение аналитического виджета на главном экране Company Workspace для ролей:

```text
OWNER / Руководитель компании
PARTNER / Партнёр
```

Речь про виджет:

```text
Аналитика за квартал
Доход
Задолженность
Переплата
Активные проекты
```

Цель: убрать временные заглушки, подключить реальные метрики на основе операций `REPORT / Отчёт` и `TRANSFER / Перевод`, а также реализовать интерактивный выбор месяца внутри графика.

## 5. Общие бизнес-термины

### 5.1 REPORT / Отчёт

`REPORT` начисляет заработанное и формирует задолженность.

REPORT не выполняет фактическую выплату.

Фактическая выплата выполняется отдельной операцией:

```text
TRANSFER / Перевод
```

### 5.2 TRANSFER / Перевод

`TRANSFER` закрывает задолженность, когда деньги переводятся участнику на личный баланс.

### 5.3 Основные формулы

```text
Лично заработано = начисления по подтверждённым REPORT
Лично получено = фактические выплаты через TRANSFER на личный баланс
Задолженность = max(Лично заработано - Лично получено, 0)
Переплата = max(Лично получено - Лично заработано, 0)
```

---

## 6. Правила для OWNER / Руководителя компании

Для Руководителя компании аналитика считается по всей активной компании.

### 6.1 Область данных OWNER

Учитывать:

```text
Все проекты компании
Все участники проектов компании
Все подтверждённые отчёты компании
Все подтверждённые переводы компании
```

### 6.2 Доход OWNER

Для Руководителя компании показатель `Доход` считается по варианту Б:

```text
Доход OWNER = прибыль компании / участников по подтверждённым REPORT
```

То есть доход — это начисленное заработанное по REPORT, включая:

```text
прибыль
наценку
начисления по подтверждённым отчётам
```

Не считать доходом OWNER:

```text
поступления от заказчиков
фактические переводы участникам
черновики отчётов
отчёты без применённых финансов
отчёты с откатанными финансовыми дельтами
```

### 6.3 Задолженность OWNER

```text
Задолженность OWNER = max(Лично заработано по участникам компании - Лично получено по участникам компании, 0)
```

Считать агрегированно по участникам проектов компании.

`Лично заработано` формируется из подтверждённых REPORT:

```text
основной получатель отчёта
получатели прибыли
получатели наценки
```

`Лично получено` формируется из TRANSFER на личный баланс.

### 6.4 Переплата OWNER

```text
Переплата OWNER = max(Лично получено по участникам компании - Лично заработано по участникам компании, 0)
```

Переплата отображается статистически и не запускает автоматический возврат денег.

### 6.5 Активные проекты OWNER

```text
Активные проекты OWNER = все активные проекты компании
```

## 7. Правила для PARTNER / Партнёра

Для Партнёра аналитика считается не просто по проектам, где он руководитель проекта, а по двум контурам:

```text
1. Контур активных проектов.
2. Финансовый контур личных начислений и выплат Партнёра.
```

### 7.1 Активные проекты PARTNER

Количество активных проектов для Партнёра:

```text
только проекты, где текущий контрагент является PROJECT_HEAD первого порядка
```

Это используется для графика и количества активных проектов.

### 7.2 Финансовый контур PARTNER

Доход / задолженность / переплата для Партнёра считаются по всем операциям, где текущий контрагент получает деньги или начисления.

Учитывать:

```text
1. Проекты, где Партнёр является PROJECT_HEAD.
2. Проекты, где Партнёр является участником первого порядка.
3. Проекты, где Партнёр автоматически стал участником второго порядка.
4. Отчёты, где Партнёр является:
   - основным получателем отчёта;
   - получателем прибыли по статье расходов;
   - получателем наценки по статье расходов.
5. Переводы, где Партнёр получает деньги на личный баланс.
```

Важно:

```text
Партнёр может быть участником второго порядка и получать процент прибыли / наценки по статье расходов.
Эти начисления должны попадать в расчёт задолженности / переплаты / дохода после перевода.
```

### 7.3 Доход PARTNER

```text
Доход PARTNER = лично получено Партнёром через TRANSFER на личный баланс
```

Учитывать все `ProjectParticipant` текущего Партнёра в рамках выбранной компании:

```text
first-order
second-order
PROJECT_HEAD
PARTNER
```

### 7.4 Задолженность PARTNER

```text
Задолженность PARTNER = max(Лично заработано Партнёра - Лично получено Партнёра, 0)
```

`Лично заработано` включает начисления из REPORT:

```text
сумма основному получателю отчёта
доля прибыли
доля наценки
```

### 7.5 Переплата PARTNER

```text
Переплата PARTNER = max(Лично получено Партнёра - Лично заработано Партнёра, 0)
```

## 9. Интерактивность столбцов месяца

### 9.1 По умолчанию

```text
selectedMonth = null
```

Отображение:

```text
Заголовок: Аналитика за квартал
Показатели: за квартал
Все столбцы без выделения
```

### 9.2 Первый клик по месяцу

Если пользователь нажимает на столбец месяца:

```text
1. Выбранный столбец визуально выделяется / затемняется.
2. Заголовок меняется:
   «Аналитика за квартал» → «Аналитика за май»
   или другой выбранный месяц.
3. Все показатели виджета пересчитываются только по операциям выбранного месяца.
```

### 9.3 Клик по другому месяцу

Если пользователь выбирает другой месяц:

```text
1. Предыдущий столбец снимает выделение.
2. Новый столбец выделяется / затемняется.
3. Заголовок меняется на новый месяц.
4. Доход / Задолженность / Переплата / Активные проекты пересчитываются по новому месяцу.
```

### 9.4 Повторный клик по тому же месяцу

Если пользователь повторно нажимает на уже выбранный месяц:

```text
1. Выбор месяца снимается.
2. Все столбцы возвращаются в обычное состояние.
3. Заголовок снова становится:
   «Аналитика за квартал»
4. Показатели возвращаются к расчёту за весь квартал.
```

## 10. Расчёт показателей за месяц

### 10.1 Доход по месяцу

Для OWNER:

```text
Доход за месяц = начисленное заработанное / прибыль / наценка по REPORT, применённым в выбранном месяце
```

Для PARTNER:

```text
Доход за месяц = TRANSFER на личный баланс Партнёра, применённый / завершённый в выбранном месяце
```

### 10.2 Задолженность и переплата по месяцу

Задолженность и переплата — это состояние, поэтому для выбранного месяца считать их на конец выбранного месяца:

```text
Заработано накопительно до конца месяца
Получено накопительно до конца месяца

Задолженность = max(Заработано - Получено, 0)
Переплата = max(Получено - Заработано, 0)
```

Для текущего квартала без выбранного месяца считать состояние на текущую дату.

### 10.3 Активные проекты по месяцу

Для OWNER:

```text
Активные проекты за месяц = активные проекты компании, которые были активны в выбранном месяце
```

Для PARTNER:

```text
Активные проекты за месяц = проекты, где Партнёр является PROJECT_HEAD первого порядка и проект был активен в выбранном месяце
```

Если в текущей модели проекта нет истории активности по месяцам — использовать текущий статус проекта и дату создания как MVP-логику, без сложной исторической реконструкции.

## 12. Защита от двойного счёта

Если один и тот же Партнёр получил несколько начислений в одном REPORT:

```text
например долю прибыли и долю наценки
```

нужно суммировать начисления по фактическим дельтам кошелька, а не делать join, который размножит строки.

Правило:

```text
Одна финансовая дельта = один вклад в расчёт.
```

## 13. Клики по показателям виджета

При клике на показатель открывается список операций, из которых сформирована выбранная цифра.

Показатели:

```text
Доход
Задолженность
Переплата
```

### 13.1 OWNER — клик по Доходу

Для Руководителя компании при клике на `Доход` открывать:

```text
Список REPORT, по которым сформирована прибыль / наценка.
```

То есть список операций `REPORT`, где финансы применены и которые участвуют в расчёте дохода компании.

Детализация по участникам:

```text
кто сколько заработал
доля прибыли
доля наценки
```

открывается уже внутри конкретного отчёта.

### 13.2 OWNER — клик по Задолженности

Открывать список REPORT, где начисления компании / участников ещё не закрыты переводами полностью.

### 13.3 OWNER — клик по Переплате

Открывать список TRANSFER / связанных операций, из-за которых получено больше, чем начислено.

Для MVP допускается открыть список операций, сформировавших переплату, с пояснением в заголовке.

### 13.4 PARTNER — клик по Доходу

Открывать список TRANSFER, по которым Партнёр получил деньги на личный баланс.

### 13.5 PARTNER — клик по Задолженности

Открывать:

```text
Список REPORT, где Партнёру начислено заработанное, но сумма ещё не полностью закрыта TRANSFER.
```

Логика:

```text
REPORT создаёт задолженность
TRANSFER закрывает задолженность
```

Поэтому в MVP точка входа — список REPORT.

### 13.6 PARTNER — клик по Переплате

Открывать список TRANSFER / REPORT-связанных операций, из-за которых у Партнёра получено больше, чем начислено.

## 18. Тесты

Добавить backend-тесты.

Минимум:

```text
1. OWNER видит доход / задолженность / переплату по всей компании.
2. PARTNER видит доход / задолженность / переплату по своим начислениям и выплатам.
3. PARTNER second-order profit share учитывается в задолженности.
4. TRANSFER на личный баланс уменьшает задолженность / увеличивает доход.
5. Если personal_received > personal_earned, появляется переплата.
6. Откат REPORT исключает начисление из задолженности.
7. Откат TRANSFER исключает полученную выплату из дохода.
8. OWNER active projects считаются по всей компании.
9. PARTNER active projects считаются только по проектам, где он PROJECT_HEAD.
10. Метрики выбранного месяца отличаются от квартальных.
11. Список операций по клику на Доход OWNER возвращает REPORT.
12. Список операций по клику на Задолженность PARTNER возвращает REPORT.
```

Если полный набор сразу слишком большой — минимум сделать 5 критичных тестов:

```text
OWNER all company
PARTNER own earned
PARTNER second-order earned
TRANSFER received affects income/debt
overpayment
```
